# PHP Framework-Specialty Skills Reference

These skills are **conditional**: `skill-forge` generates one only when the profile's section **3.1 (Framework-Specialty Signals)** marks the matching signal `confirmed` or `inferred` (never for `unknown`/`none`). Each row below generalizes a pattern that the Laravel and/or Symfony accelerators in this monorepo implement as a *framework-named* skill (e.g. `eloquent`, `messenger-designer`) into a *signal-named* skill that fits whatever ORM/queue/admin-panel/etc. the target actually uses - Eloquent, Doctrine, a bare PDO repository layer, RabbitMQ, SQS, or anything else the evidence points to. Never name a generated skill after a framework the target does not use; name it after the generic concept and ground its body in the target's real package/tool.

Unlike the 14 process skills (`references/php-process-skills.md`), these are NOT generated by default - each requires its own evidence line in section 3.1.

## The Specialty Catalog

| Generic skill name | Detection signal (section 3.1) | What it guides | Generalizes |
| --- | --- | --- | --- |
| `orm-patterns` | An ORM is in use (Eloquent, Doctrine, or another) beyond basic CRUD. | **How application code uses the schema through the ORM**, assuming `database-designer` already designed that schema: relationships/associations, accessor/mutator or cast patterns, query scopes/specifications, lifecycle hooks (model events/observers or Doctrine listeners), mass-assignment/hydration safety, and efficient loading for large datasets (eager loading / batching) to avoid N+1s. Does NOT cover table/column/index/migration design - that stays in `database-designer` (`php-frameworks.md`'s "Design & Interaction Skills"); link to it rather than re-deriving schema design here. | Laravel `eloquent`; the ORM-specific portions of Symfony's `database-designer`. |
| `migration-safety` | A schema-migration tool is present (Laravel/Doctrine migrations, Phinx, or another) with a non-trivial migration history. | Safe **rollout** of a migration once `database-designer` has decided what the schema change should be: additive-first rollout order (add -> backfill -> constrain -> clean up), online-safe index/lock strategy, rollback limitations, and verification steps before a migration ships. Owns deployment-safety only, not what to design - see `database-designer` (`php-frameworks.md`) for the schema-design decision itself. | Symfony `doctrine-migration-designer`. |
| `async-jobs` | A background-work mechanism is wired (queued jobs, a message bus/broker, or scheduled async work) - whether via a first-party abstraction (Laravel queues, Symfony Messenger) or a third-party broker already listed in section 4. | Reliable async work: job/message anatomy, idempotency/deduplication, retry and failure-transport strategy, transaction boundaries around dispatch, and overlap/duplicate-run prevention for scheduled work. | Laravel `queues-jobs` + `console-scheduler`'s scheduling half; Symfony `messenger-designer`. |
| `event-boundary-review` | Domain or framework events with listeners/subscribers/observers are present. | Keeping event listeners/subscribers as thin adapters that delegate real business workflow to services, rather than hiding business logic inside a listener; covers both domain pub/sub and framework-event hooks. | Laravel `events-notifications`'s event half; Symfony `event-subscriber-designer`. |
| `notification-delivery` | Multi-channel outbound user notifications are wired (mail/database/broadcast/SMS/chat notification classes), distinct from a plain transactional-mail integration already in section 4. | Multi-channel notification design: channel selection per notification, templating, delivery-failure handling, and avoiding duplicate sends. | Laravel `events-notifications`'s notification half. |
| `caching-strategy` | An in-app caching layer is used beyond what's already captured as a section-4 integration (cache-aside usage, tags, model/query-level caching). | **Cache-CORRECTNESS**, not raw speed: what to cache, key/tag design, stampede prevention, and - most importantly - invalidate-on-write correctness so stale reads never outlive a write. Owns this depth exclusively - the universal `performance` skill (`php-frameworks.md`) only names caching as one possible hot-path lever and links here for the correctness contract; it must not re-derive it. | Laravel `caching`. |
| `file-storage` | A filesystem/object-storage abstraction is used for uploads or generated files (local disk, S3, or another Flysystem-style adapter), distinct from the raw integration entry in section 4. | Secure upload handling, disk/visibility configuration (public vs. private), signed/temporary URL generation, and safe streaming of large files. | Laravel `file-storage`. |
| `auth-scaffolding` | Session/cookie-based authentication and/or an authorization-policy layer (Policies/Gates, Symfony Voters, or equivalent) is present, distinct from the token/OAuth integration already in section 4/6. | Authentication setup (guards, starter-kit wiring) and object-level authorization: policy/voter design, allow/deny/abstain logic, and the call sites that must enforce it. | Laravel `auth-scaffolding`; Symfony `security-voter-designer`. |
| `form-validation-design` | Structured input validation beyond simple inline rules - a dedicated Form/Request/DTO + Validator layer with custom constraints or validation groups. | Choosing and implementing the validation architecture: custom constraint/rule classes, validation groups/scenarios, and consistent invalid-payload/error-response shape. | Symfony `form-validator-designer`; the validation-architecture portion of Laravel/PHP Core's `api-designer`. |
| `admin-panel` | A back-office/admin-panel package is wired (e.g. an Eloquent-CRUD admin generator, an EasyAdmin/Sonata-style bundle, or similar) for internal/staff-facing CRUD. | Building/extending admin-panel resources: CRUD forms and tables bound to the real domain models, relation managers, custom actions/widgets - scoped explicitly to internal tooling, not customer-facing UI. | Laravel `filament`. |
| `api-platform-design` | A declarative, metadata-driven API resource framework is wired (e.g. API Platform or a similar attribute/annotation-driven REST layer). | Designing declarative API resources: operation/metadata configuration, input/output DTOs, providers/processors that delegate to real services/repositories, and per-operation security expressions. **Scope vs. `api-designer` (`php-frameworks.md`):** when this framework is the target's primary/sole API mechanism, `api-platform-design` owns ALL resource-level design and `api-designer` narrows to any remaining hand-rolled endpoints only (stating so explicitly) rather than both re-deriving the same route/contract guidance. | Symfony `api-platform-designer`. |
| `console-commands` | The target ships custom CLI commands (beyond the framework's own built-ins) for operational or batch tasks. | Implementing CLI commands: input/argument validation, delegating business logic to a service (never inlining it in the command), safe progress/output streaming, and correct exit codes. | Symfony `console-command-coder`; the command-authoring half of Laravel `console-scheduler`. |
| `repository-review` | A repository/data-access layer sits between domain code and persistence (ORM repositories, custom query classes, or a Doctrine/PDO repository pattern). | Auditing data-access code: parameter binding safety, business-readable method names, index/sort alignment, bounded/stable pagination, N+1 risk, and keeping repositories free of authorization/formatting/side-effect logic. | Symfony `repository-reviewer`. |
| `container-review` | Dependency injection is configured declaratively (a container config file such as `services.yaml`/`services.php`, autowiring, compiler passes), not purely via code-driven bindings. | Reviewing DI wiring for correctness: autowiring/alias/tag configuration, decorator ordering, compiler-pass effects, and verifying actual wiring with the framework's own container-debug tooling where available. | Symfony `container-reviewer`. |
| `test-data-factories` | Deterministic test-data helpers exist or are needed (model factories, fixture classes, object-mother builders). | Building realistic, overrideable, non-random-by-default test-data factories/fixtures, with no assertions embedded in factory code. | Laravel's factory conventions (folded into `eloquent`/`test-generator`); Symfony `fixture-factory-generator`. |
| `package-authoring` | The target itself is a distributable Composer package/library (a reusable dependency, not a deployed application) - e.g. no deployment target in section 5, a library-shaped `composer.json` `type`, and consumers expected to `require` it. | Package-authoring concerns: public API stability, Service-Provider/Bundle-style integration points for the consuming app, publishable config/migrations/views, and package-local testing (e.g. Orchestra Testbench or an equivalent harness). | Laravel `package-developer`. |

## Generation Rule

For each row above whose signal in profile section 3.1 is `confirmed`/`inferred` **present** (the pattern itself was found), generate exactly one skill named per the "Generic skill name" column (or a close, evidence-fitting variant, e.g. `eloquent-patterns` instead of `orm-patterns` when Eloquent specifically is confirmed - keep it descriptive of the real tool). Ground the body in the target's real package/class names from the scan, never in the framework example named in "Generalizes" unless that IS the target's actual framework. A confidence tag on a line does NOT by itself mean "generate this skill" - a line can be `confirmed`/`inferred` **absent** (e.g. "Repository/data-access layer: inferred none", "DI container configuration style: code-driven bindings only (confirmed)" when the row requires declarative config, "Package vs. application: deployed application (confirmed)" when the row requires a distributable package) - those are confirmed/inferred findings of *absence* and MUST be skipped exactly like `unknown`/bare `none`. Only generate for a signal whose confirmed/inferred finding is that the pattern is actually present.
